# ADR-0001: design.md v0.1 を本プロジェクトの設計の単一の正として採択する

- **ステータス**: accepted
- **日付**: 2026-06-10
- **関連**: docs/design.md 全体 / §0.3(管理ルール)・§12(開発規約) / Phase 0
- **備考**: 本 ADR は design.md §11 Phase 0 で起票が予定されていた未起票 ADR。
  採択(`accepted`)は人間レビューで行う。

## 背景

社内ナレッジ管理プラットフォーム `stratum` の設計は `docs/design.md` に集約されている。
このドキュメントはアーキテクチャ・データ設計・コンポーネント責務・実装ロードマップ・開発規約を
網羅し、コーディングエージェントと人間の双方にとっての判断基準となることを意図している。

design.md §0.3 は「設計変更は必ず ADR を起票してから本ドキュメントへ反映する」と定めるが、
その前提となる「design.md 自体を設計の単一の正とする」という最初の決定そのものが ADR として
記録されていない。この基底の決定を明文化し、以降の全 ADR・全実装が design.md を参照根拠として
機能できる状態を確立する。

## 決定

### D1. design.md v0.1 を設計の単一の正(single source of truth)とする

`docs/design.md`(以下 v0.1)を本プロジェクトの設計上の唯一の正とする。実装・レビュー・タスク
分割の判断は v0.1 と関連 ADR に基づいて行い、設計と実装が食い違う場合は実装を進めず人間に確認を
求める(§12.2)。

### D2. 設計変更のプロセスを ADR 先行とする

design.md §0.3 / §12.2 のとおり、アーキテクチャ・データスキーマ・外部サービス・権限方針に関わる
変更は、コードや design.md 本体の編集より先に `docs/adr/` へ ADR ドラフト(`status: proposed`)を
起票する。design.md 本体は保護パス(`.claude/hooks/guard-protected-paths.sh`)であり、本文への
転記は人間レビューを経て行う。ADR と design.md の差分が残る間は ADR を優先根拠とする。

### D3. v0.1 を凍結基準とし、改訂は版管理する

本 ADR の採択時点の design.md を v0.1 として確定する。以降の改訂は ADR 経由で行い、ドキュメント
冒頭の版表記を更新する。過去版との差分は Git 履歴で追跡する。

### 付随する取り決め(確認のみ・新規決定ではない)

- §2 の設計原則 P1〜P8 を全コンポーネント共通の制約とする。
- §1.3 の非目標(汎用 Wiki 置換・事前オントロジー設計・機微情報取り込み・社外公開・議事録生成
  再実装・リアルタイム音声処理)は実装提案の対象外とする。
- 既に `accepted` 済みの ADR-0003(kb-core スキーマ確定)は v0.1 を前提に成立している。

## 影響

- 以降の全 ADR は本 ADR を暗黙の前提として「design.md の単一正」に依拠できる。
- design.md の改訂手順(ADR 先行 → 人間レビュー → 本文反映)が正式に発効する。
- 採択後、design.md 冒頭に v0.1 採択を ADR-0001 として参照する追記が望ましい(人間レビューで対応)。

## 却下した代替案

- ADR を起票せず design.md を暗黙の正として運用する案 → 設計の正がどの時点で確定したかを後から
  追跡できず、§0.3 の「ADR を起票してから反映する」運用と矛盾するため却下。
- 設計を ADR 群のみで管理し統合ドキュメントを持たない案 → 全体像の把握とオンボーディングの
  コストが高く、バス係数低減という本プロジェクトの目的(§1.2)に反するため却下。
