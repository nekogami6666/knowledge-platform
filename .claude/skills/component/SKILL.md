$ARGUMENTS で指定されたコンポーネントを実装する。次を順に、人間への確認なしで進める。

1. planner サブエージェントに、該当コンポーネントの実装計画を立てさせる
2. 計画の「設計上の論点」を確認する。design.md の解釈が分かれる重大な論点が
   あれば、実装に入らず論点を整理して止まる。なければ計画に従って実装する
3. design.md の受け入れ条件をテストとして表現する
4. pnpm typecheck / pnpm lint / pnpm test が全て green になるまで反復する
5. green の証拠（コマンド出力）と、テストがカバーするケース一覧を提示する
6. design-reviewer サブエージェントに diff のレビューをさせる。
   ブロッカー・要修正は対応して再レビュー。対応しない所見は反論理由を付けて
   PR 本文の「レビュー済み所見」欄に残す
7. feat/<コンポーネント名> ブランチを切り、conventional commits でコミットする
8. コミットメッセージと PR 本文（変更概要・受け入れ条件との対応・レビュー済み所見・
   レビュー観点）の案を提示して止まる。push と PR 作成は人間の承認を待つ

制約:
- スコープは指定された §6 節の受け入れ条件まで。隣接コンポーネントに手を出さない
- 設計と矛盾を見つけたら、実装で勝手に解決せず、論点を整理して止まる
- 新規の外部依存は、追加前に理由を提示して止まる
